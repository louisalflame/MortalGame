# 文檔撰寫指導原則 - MortalGame

## 🎯 核心原則：濃縮精要，聚焦設計思路

### 文檔定位與目標
**文檔是設計思路的濃縮概要，而非程式碼的逐行解說**
- 📖 閱讀者需要快速理解系統的**業務邏輯**和**設計框架**
- 🔍 詳細程式碼實作可透過檔案路徑直接查看，無需在文檔中重複
- 🧠 重點在於解釋**為什麼這樣設計**，而非**怎麼實作的**

---

## ✅ 文檔應該包含的內容

### 1. 宏觀設計框架
- **系統架構**：整體設計思路與模式
- **職責分離**：各組件的職責邊界
- **數據流向**：資料在系統中的流動路徑
- **擴展點**：系統的可擴展性設計

### 2. 業務邏輯流程
- **核心流程**：主要業務流程的步驟概要
- **決策點**：關鍵的邏輯判斷與分支
- **狀態轉換**：系統狀態變化的觸發條件
- **異常處理**：錯誤情況的處理策略

### 3. 設計要點與考量
- **設計模式**：使用的設計模式及其原因
- **權衡取捨**：設計決策的考量因素
- **約束條件**：設計時面臨的限制
- **演進方向**：未來可能的發展方向

### 4. 精確QA環節
透過關鍵問題深化理解：
```
🔍 設計思路類問題：
- 為什麼選擇這種架構模式？
- 這樣設計解決了什麼核心問題？
- 系統間的耦合如何控制？

🔄 流程邏輯類問題：
- 核心業務流程的關鍵步驟是什麼？
- 資料流向的設計考量是什麼？
- 異常情況如何優雅處理？

🚀 擴展性類問題：
- 系統如何應對需求變化？
- 新功能的加入點在哪裡？
- 哪些部分是可插拔的？
```

## ❌ 文檔應該避免的內容

### 1. 程式碼直接貼入
- **避免**：大段程式碼貼入文檔
- **替代**：提供檔案路徑 `[ClassName](path/to/file.cs)` 讓讀者直接查看

### 2. 逐行程式碼解說
- **避免**：「第X行做什麼，第Y行做什麼」的流水帳式解說
- **替代**：解釋方法的**目的**和**在整體流程中的角色**

### 3. 實作細節羅列
- **避免**：詳細的演算法實作、資料結構細節
- **替代**：說明**為什麼選擇這種實作方式**的設計考量

### 4. API 使用說明書
- **避免**：像 API 文檔一樣列出所有方法參數
- **替代**：說明**核心介面的設計意圖**和**使用場景**

## 🚨 準確性標準
**所有文檔內容必須基於實際存在的程式碼，禁止創造虛構的實作細節**

### 內容分類
1. **設計分析**：基於實際程式碼結構的設計思路分析 ✅
2. **業務流程**：基於實際邏輯的業務流程概要 ✅
3. **架構觀察**：基於實際系統的架構模式觀察 ✅
4. **虛構實作**：不存在的程式碼細節或API ❌

## 📁 資料夾結構筆記整理方針

### 階層化文檔組織
基於腳本檔案的資料夾結構建立對應的文檔系統，採用**由細到粗**的分層整理方式：

### 組織原則
1. **子資料夾文檔**：對每個包含腳本的資料夾建立同名 `.md` 檔案
2. **逐層整理**：從最深層資料夾開始，逐步向上整理
3. **宏觀概要**：上層文檔以更宏觀的視角概括下層內容

### 文檔建立規範

#### 子系統層文檔（葉子節點）
**對象**：包含具體腳本檔案的資料夾
**檔名**：`{資料夾名稱}.md`
**內容**：該資料夾下所有腳本的**整體功能概要**

#### 系統層文檔（中間節點）
**對象**：包含子資料夾的資料夾
**檔名**：`{資料夾名稱}.md`  
**內容**：以更**宏觀的系統視角**概括所有子系統的功能

#### 架構層文檔（根節點）
**對象**：最頂層的系統資料夾
**檔名**：`{頂層系統名稱}.md`
**內容**：從**架構設計角度**統整所有子系統的協作關係

### 撰寫重點
- **子系統層**：專注於該資料夾內腳本的**共同功能領域**
- **系統層**：著重於**子系統間的協作關係**和**整體設計思路**
- **架構層**：強調**系統在整個專案中的角色**和**與其他系統的依賴關係**
